## Introduction

**FlowG** favors **availability** and **partition tolerance** over strong
consistency (it is an "AP" system, see [Consensus](./consensus)). Every node
accepts writes locally and the cluster converges towards a shared state over
time — a model known as **eventual consistency**.

The foundation that makes this safe is described in
[How Data Is Replicated?](/docs/design/data-replication): every record is stored
in a **Last-Writer-Wins envelope** stamped with a **Hybrid Logical Clock**
timestamp. Because conflict resolution is deterministic and order-independent,
nodes can exchange records in any order, at any time, and still converge.

**FlowG** has 3 replicated datasets:

 - one for authentication and permissions
 - one for configuration (pipelines, transformers, ...)
 - one for the actual logs

Replication is achieved through **two complementary mechanisms**:

```mermaid
flowchart TD
  W[Local write committed] --> B
  subgraph FP["Fast path — low latency"]
    B[Broadcast records to peers]
  end
  subgraph SN["Safety net — guaranteed convergence"]
    A[Periodic anti-entropy reconciliation]
  end
  W -.-> A
  B --> P1[(Peer stores)]
  A --> P1
```

 - **Broadcast** propagates fresh writes immediately, for low latency.
 - **Anti-entropy** periodically reconciles nodes, catching anything a broadcast
   may have missed.

Neither mechanism needs the other to be correct: broadcast makes replication
*fast*, anti-entropy makes it *guaranteed*.

## Broadcast (fast path)

When a node commits a write, it immediately gossips the freshly written records
to the rest of the cluster. Peers merge them into their local store using the
Last-Writer-Wins rule. This keeps replication latency low under normal
conditions.

```mermaid
sequenceDiagram
  participant C as Client
  participant N1 as Node 1
  participant N2 as Node 2
  participant N3 as Node 3

  C->>N1: Write
  N1->>N1: Store locally (envelope + HLC)
  N1-->>C: Acknowledge
  par Broadcast to peers
    N1-->>N2: New records
    N2->>N2: LWW merge
  and
    N1-->>N3: New records
    N3->>N3: LWW merge
  end
```

Broadcast is **best-effort**: a message may be lost, or a node may be temporarily
unreachable (for example during a network partition). On its own it is therefore
not enough to guarantee that every node eventually holds every record — which is
why it is paired with anti-entropy.

## Anti-entropy (safety net)

Periodically, each node reconciles its datasets with the other members of the
cluster. To avoid re-sending data that a peer already has, every node keeps track
— for each peer and each dataset — of **how much of that peer's data it has
already received**. We call this position a **watermark**.

During reconciliation, nodes gossip their watermarks to one another. When a node
sees — from a peer's watermark — that the peer is missing some of the records
**it** holds, it **pushes** that missing tail to the peer. This makes
synchronization **incremental**: steady-state reconciliation is cheap, and a node
that has fallen behind (or just joined) receives exactly the tail of history it is
missing — starting from the very beginning on first contact.

```mermaid
sequenceDiagram
  participant N1 as Node 1
  participant N2 as Node 2

  Note over N1,N2: Periodic reconciliation
  N2->>N1: Watermark — how much of N1's data N2 already holds
  N1->>N1: Select its records newer than that watermark
  N1-->>N2: Push the missing tail of records
  N2->>N2: LWW merge
  N2->>N2: Advance its watermark for N1
```

Because merges are conflict-free, a watermark only needs to be an *optimization*:
even if a record is delivered more than once, applying it again is harmless.

### Skipping caught-up datasets

The watermarks also let a node avoid talking to a peer that has nothing to learn.
Before pushing to a peer, a node compares — for each dataset — the watermark the
peer reports against its own latest local version:

 - if the peer's watermark already covers the latest version, the dataset is
   **up to date** and is skipped entirely;
 - if *every* dataset is up to date, no exchange happens at all.

This means that, once a cluster has converged, periodic reconciliation becomes
essentially free: nodes confirm they are in sync from the lightweight state they
already piggyback on each other, and only generate real traffic when a dataset
has actually moved forward.

```mermaid
flowchart TD
  R[Reconciliation due] --> L{For each dataset}
  L --> C{Peer's watermark<br/>covers our latest?}
  C -- Yes --> SK[Skip — nothing to push]
  C -- No --> RQ[Push records<br/>after the watermark]
  SK --> D{Any dataset left to push?}
  RQ --> D
  D -- No --> Q[Stay silent]
  D -- Yes --> X[Push & merge]
```

### Recovering from a partition

Anti-entropy is what allows the cluster to heal after nodes have been isolated
from each other. While partitioned, each side keeps accepting writes locally.
When connectivity is restored, the next reconciliation exchanges everything that
diverged, and Last-Writer-Wins deterministically settles any conflicting updates.

```mermaid
sequenceDiagram
  participant N1 as Node 1
  participant N2 as Node 2

  rect rgb(255, 243, 212)
    Note over N1,N2: Network partition
    N1->>N1: Accept writes locally
    N2->>N2: Accept writes locally
  end

  Note over N1,N2: Connectivity restored
  N1->>N2: Watermarks
  N2-->>N1: Records written during the partition
  N1->>N2: Records written during the partition
  Note over N1,N2: Both sides converge (LWW)
```

### The safe-recovery window

Healing after a partition relies on **tombstones** still being present on the
nodes that stayed up (see
[Deletions and tombstones](/docs/design/data-replication#deletions-and-tombstones)).
A tombstone is what lets a deletion win against an older value when the two sides
reconcile. But tombstones cannot be kept forever — once every node has converged
they are garbage-collected to reclaim space. This retention period is the
**grace period**, and it defines a hard limit on how long recovery stays safe.

Consider a node that is isolated for *longer* than the grace period. Meanwhile,
on the healthy side of the cluster, some keys are deleted **and** their
tombstones are eventually garbage-collected. When the straggler finally
reconnects, it still holds the old values, but there is no longer any tombstone
to out-rank them. Naively pushing its data back would **resurrect** those deleted
keys across the whole cluster.

To prevent this, a node that has been out of contact longer than the grace period
is considered **too stale to be trusted as a source**. Instead of pushing its
diverged state, it **rebuilds its datasets from scratch** from a peer that stayed
healthy, **discarding any writes it made while isolated beyond the window**. This
trades a straggler's offline changes for the guarantee that a deletion is never
undone.

```mermaid
flowchart TD
  R[Node reconnects] --> Q{Out of contact<br/>longer than the<br/>grace period?}
  Q -- No --> N[Normal reconciliation<br/>keeps local writes]
  Q -- Yes --> H{A healthy peer<br/>is reachable?}
  H -- Yes --> B[Rebuild from that peer<br/>discard offline writes]
  H -- No --> W[Wait for a healthy peer]
```

:::warning Operational sizing
The grace period is the **maximum downtime or partition a node can survive
without losing the writes it made while isolated**. Operators should set it
**longer than the longest outage they want to tolerate** for a single node.
:::

A whole-cluster restart is a special case: if *every* node comes back stale at the
same time, there is no healthy peer to rebuild from — but their on-disk tombstones
are still intact, so they simply resume normal reconciliation among themselves
instead of waiting indefinitely.

## Putting it together

The diagram below summarizes how a single write reaches the rest of the cluster
through both paths:

```mermaid
flowchart LR
  W[Write on a node] --> S[(Local store)]
  S -->|immediately| BC[Broadcast]
  S -->|periodically| AE[Anti-entropy]
  BC --> M[Peers merge via LWW]
  AE --> M
  M --> CV([Cluster converges])
```

The concrete HTTP endpoints used to carry broadcasts and synchronization batches
between nodes are documented in the [Architecture](./architecture) page.
